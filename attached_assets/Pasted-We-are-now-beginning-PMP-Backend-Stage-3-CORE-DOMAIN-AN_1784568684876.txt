==================================================
STAGE 3 OBJECTIVE
==================================================

Create the core PMP marketplace domain.

The fundamental concept is:

A Clerk-authenticated person has a PMP identity.

That identity may participate in the marketplace as:

1. A provider
2. An employer
3. Potentially both in the future

IMPORTANT ARCHITECTURAL RECOMMENDATION:

Do not permanently hard-code a person into only one marketplace identity.

Authentication identity and marketplace participation must remain separate.

A single PMP user should be able to have:

- A provider profile
- An employer profile

without creating multiple Clerk accounts.

However, do not create unnecessary complexity if the existing architecture strongly suggests another approach. Inspect the current system and document the decision before implementation.

Design this carefully.

==================================================
PROVIDER PROFILE
==================================================

The provider system must support two provider types:

ARTISAN

An artisan is someone whose work primarily involves practical, physical, technical, or hands-on skills.

Examples may include:

- Plumber
- Electrician
- Carpenter
- Welder
- Tailor
- Mechanic
- Hair stylist
- Makeup artist
- Builder
- Mason
- Technician

PROFESSIONAL

A professional is someone whose work primarily operates in professional, corporate, technical, creative, academic, or specialized knowledge fields.

Examples may include:

- Software developer
- Accountant
- Lawyer
- Designer
- Data analyst
- Marketing professional
- Project manager
- Consultant
- Architect
- Engineer

Do not assume that the distinction between artisan and professional determines quality.

It is primarily a classification and discovery mechanism.

The database design should allow future expansion of provider types without requiring a complete schema rewrite.

==================================================
CATEGORIES AND SKILLS
==================================================

Create a structured reference-data foundation.

Categories and skills should not be arbitrary unstructured strings everywhere.

The system should be able to support:

Category
    └── Skills

Example:

Technology
    ├── Software Development
    ├── UI/UX Design
    ├── Cybersecurity
    ├── Data Analysis
    └── Cloud Engineering

Skilled Trades
    ├── Plumbing
    ├── Electrical Work
    ├── Carpentry
    ├── Welding
    └── Masonry

Creative Services
    ├── Photography
    ├── Videography
    ├── Graphic Design
    └── Fashion Design

The initial categories and skills should be treated as seed/reference data.

Do not create an unmaintainable hardcoded system.

Design this so that the operations/admin system can eventually manage reference data.

==================================================
PROFILE COMPLETENESS
==================================================

Create a profile completeness foundation.

Important:

PROFILE COMPLETENESS IS NOT PROFESSIONAL QUALITY.

A profile with 100% completion does not automatically mean that the provider is better.

The completeness system should only measure whether relevant profile information has been provided.

For example:

- Basic identity
- Headline
- About section
- Category
- Skills
- Experience
- Location
- Availability
- Portfolio references where applicable

The scoring system should be:

- Explainable
- Deterministic
- Testable
- Easy to change later

Do not build the final provider ranking algorithm in Stage 3.

The future ranking system will be implemented in Stage 5.

Do not use profile completeness as a substitute for ranking quality.

==================================================
DATABASE DESIGN REQUIREMENTS
==================================================

Before writing migrations:

1. Inspect the existing schema.
2. Identify the existing users table.
3. Identify the existing roles and permissions tables.
4. Identify the existing Clerk identity mapping.
5. Design the relationships.
6. Check for duplication.
7. Check for future migration problems.

Then document the proposed Stage 3 schema before implementing it.

The schema must:

- Use PostgreSQL-compatible types
- Use foreign keys where appropriate
- Use unique constraints where appropriate
- Use indexes for likely lookup paths
- Use timestamps consistently
- Use soft deletion only where the existing architecture requires it
- Avoid unnecessary polymorphic complexity
- Avoid storing binary files in PostgreSQL
- Avoid storing secrets in domain tables
- Avoid duplicate user records

Pay particular attention to:

- User → Provider Profile relationship
- User → Employer Profile relationship
- Provider → Categories
- Provider → Skills
- Provider → Experience
- Provider → Certifications
- Provider → Portfolio references

==================================================
AUTHORIZATION
==================================================

The existing server-side authorization architecture is authoritative.

A user must only be able to modify their own profile unless a future administrative permission explicitly allows otherwise.

For example:

User A
→ Can modify User A's provider profile.

User A
→ Cannot modify User B's provider profile.

Frontend visibility is not security.

All authorization must be enforced on the backend.

Use the existing Clerk authentication and PMP identity system.

Do not create another authorization system.
