==================================================
OBJECTIVE
==================================================

Build the backend search and provider discovery system for Precious Market Place (PMP).

The main use case is:

Employer searches for a service or type of provider
        ↓
PMP finds matching providers
        ↓
Providers are filtered
        ↓
Providers are ranked
        ↓
Results are returned in a useful order

The system must support both:

1. ARTISANS
2. PROFESSIONALS

The search system must be designed for:

- Fast search
- Accurate filtering
- Explainable ranking
- Future scaling
- Future caching
- Future search engine integration if needed

==================================================
IMPORTANT RANKING PRINCIPLE
==================================================

The frontend must NOT calculate ranking.

Ranking is backend-owned.

The frontend must render providers in the exact order returned by the backend.

The ranking system must be deterministic and testable.

Do not create a black-box ranking system that cannot be explained.

==================================================
SEARCH INPUTS
==================================================

Support searching by:

- Provider name
- Headline
- About/profile text
- Category
- Skill
- Location
- Provider type
- Verification status
- Availability

The API should support keyword search.

Example:

GET /v1/search/providers?q=web developer

Potential matches may include:

- Web Developer
- Frontend Developer
- Software Engineer
- React Developer

Search should use the existing provider profile, category, skill, experience, certification, portfolio, location, and verification data.

==================================================
RANKING SYSTEM
==================================================

Create a centralized ranking service.

Do not scatter ranking calculations across route handlers.

Conceptually:

Search Query
     ↓
Candidate Retrieval
     ↓
Filtering
     ↓
Ranking Service
     ↓
Paginated Results

The ranking system should consider relevant factors such as:

1. Text relevance
2. Category match
3. Skill match
4. Provider type match
5. Verification status
6. Profile completeness
7. Experience
8. Certifications
9. Portfolio strength
10. Availability
11. Location relevance where applicable

The exact scoring model must be:

- Centralized
- Deterministic
- Testable
- Documented

Use a weighted scoring approach.

Do not make "verified" the only ranking factor.

Do not guarantee that paying providers will automatically rank higher.

The future subscription system may support promoted recommendations, but this must be implemented as a separate, transparent ranking signal rather than secretly corrupting organic relevance.

==================================================
RECOMMENDED RANKING ARCHITECTURE
==================================================

Use separate scoring components.

Conceptually:

finalScore =
    textRelevanceScore
  + categoryMatchScore
  + skillMatchScore
  + verificationScore
  + completenessScore
  + experienceScore
  + certificationScore
  + portfolioScore
  + availabilityScore
  + locationScore
  + promotionScore (future)

Each component should be explainable.

Do not necessarily expose the complete internal score publicly.

However, the system should internally be able to explain why a provider ranked highly.

Example internal ranking explanation:

{
  "matchedSkills": 3,
  "categoryMatch": true,
  "verified": true,
  "completenessScore": 92,
  "experienceScore": 78
}

This may be used later for analytics and debugging.

==================================================
SEARCH IMPLEMENTATION
==================================================

Start with PostgreSQL-compatible search.

Do not prematurely introduce Elasticsearch, Algolia, Meilisearch, or another external search engine.

Use the existing PostgreSQL database and schema.

The implementation should be structured so a future search provider can replace the database search layer without rewriting the ranking domain logic.

Conceptually:

SearchProvider
    │
    ├── PostgreSQLSearchProvider
    │
    └── Future:
        ├── MeilisearchProvider
        ├── AlgoliaProvider
        └── ElasticsearchProvider

Do not implement the future providers now.

==================================================
SORTING
==================================================

Support a controlled set of sorting options.

Examples:

- relevance
- newest
- completeness
- experience

Do not accept arbitrary SQL sort fields directly from users.

Use an allowlist.

The default should be relevance.

If no search query is provided, the backend should return a sensible discovery order rather than attempting text relevance against an empty query.

==================================================
PAGINATION
==================================================

Use a scalable pagination strategy.

Review the existing API contracts before choosing the exact approach.

If the existing contract requires page/limit, preserve that contract.

Ensure:

- Maximum page size is enforced.
- Invalid page sizes are rejected or safely normalized.
- Results are stable.
- Total counts are handled efficiently.

Do not fetch every provider into application memory and rank thousands of records in Node/Bun if PostgreSQL can perform the filtering efficiently.

However, keep ranking logic centralized and testable.

If the first implementation uses a bounded candidate set before application-level ranking, document the tradeoff clearly.

==================================================
CACHING PREPARATION
==================================================

Do not implement a full Redis cache in this stage unless the existing architecture already requires it.

However, design the search service so caching can later be added around:

- Popular searches
- Reference data
- Stable discovery queries

Do not make the search logic dependent on cache availability.
