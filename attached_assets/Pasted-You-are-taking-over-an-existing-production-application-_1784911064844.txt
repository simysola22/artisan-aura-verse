You are taking over an existing production application called PMP (Precious Market Place).

IMPORTANT: This is an existing application with many working features. Do NOT randomly rewrite, replace, refactor, or redesign existing architecture. Your first responsibility is to understand what already exists and extend it safely.

CURRENT PRODUCTION ARCHITECTURE:
- Frontend: Vercel
- Backend: Render
- Database: Railway PostgreSQL
- Authentication: Clerk
- Frontend: React + TanStack Start/Router + TanStack Query
- Backend: Bun + Hono
- Database: PostgreSQL + Drizzle ORM
- Validation: Zod
- Existing RBAC/permissions system
- Existing operations/admin routes
- Existing audit logging
- Existing subscription/billing system
- Existing messaging system
- Existing Clerk authentication and identity synchronization

CURRENT IMPORTANT RULE:
Messaging is subscription-gated. A user must have an active subscription to create a conversation or send a message. The backend is the source of truth. Do not remove or weaken this rule.

PHASE 3 — STAFF LIFECYCLE
==================================================

The staff system should support lifecycle states such as:

- invited
- active
- suspended
- disabled
- removed

A staff member must have:

- A real individual Clerk identity
- An internal PMP identity
- One or more explicit staff roles
- Explicit permissions
- A recorded lifecycle state

Avoid shared employee accounts wherever possible.

If an employee leaves:

1. Admin suspends or disables the staff account.
2. Existing sessions are revoked if possible.
3. Future backend requests are blocked.
4. The action is recorded in the audit log.
5. The staff member cannot regain access without explicit reactivation.

PHASE 6 — SPECIAL ACCESS CREDENTIALS
==================================================

Do NOT create permanent shared admin passwords or permanent raw keys.

If emergency credentials are needed, they must be:

- Rotatable
- Revocable
- Expirable
- Individually identifiable
- Audited on every use
- Stored securely
- Never stored in plaintext in the database

Prefer storing only a secure hash of a credential.

The raw credential should be shown only at creation/rotation time if the architecture requires it.

Possible emergency credential categories:

- SUPER_ADMIN_EMERGENCY
- ADMIN_EMERGENCY
- SYSTEM_ENGINEER_EMERGENCY
- MAINTENANCE_EMERGENCY

However, do not implement these as a second hidden authentication system that bypasses Clerk unless the security architecture has been carefully designed and reviewed.

Normal staff access should use individual authenticated identities.

Emergency access should be:

1. Explicitly activated.
2. Short-lived where possible.
3. Logged.
4. Revocable.
5. Rotatable.
6. Restricted to the minimum necessary privilege.

If an emergency credential is compromised:

Admin/Super Admin must be able to rotate/revoke it without a database migration or code deployment.

PHASE 7 — ADMIN POWER
==================================================

The Admin should have broad operational control through the app UI.

The Admin should be able to perform the work of staff roles when necessary.

For example, if a verification worker is unavailable, the Admin should be able to perform verification operations.

If support staff are unavailable, the Admin should be able to perform support operations.

If a maintenance worker is removed, the Admin should be able to perform permitted maintenance operations.

However:

- All Admin actions must be audited.
- The actor must still be identified as Admin.
- The system must not pretend the Admin was the original staff member.
- The audit log must preserve the actual actor.

The Super Admin should have the highest level of control, including:

- Admin management
- Staff management
- Role management
- Permission management
- Emergency credential rotation
- Security control
- System override where explicitly designed

Do not expose dangerous operations without appropriate confirmation and, where appropriate, re-authentication.
