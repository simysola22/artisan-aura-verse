PHASE 2 — DESIGN
==================================================

Before implementation, design the Stage 7 architecture.

The design must cover:

A. CONVERSATIONS

Determine the correct conversation model.

A conversation should have an appropriate representation of:

- conversation ID
- participants
- creation timestamp
- last activity
- last message preview if appropriate
- unread state/count if appropriate
- archived/deleted state if required
- moderation state if required

Ensure the data model prevents duplicate direct conversations where appropriate.

Use database constraints where possible.

B. MESSAGES

Design messages with appropriate support for:

- message ID
- conversation ID
- sender identity
- message content
- timestamps
- edit/delete state if supported
- moderation state
- read/delivery state if appropriate

Do not expose internal database rows directly.

Use DTOs.

C. AUTHORIZATION

Messaging must use the existing Clerk + PMP identity + PMP authorization architecture.

A user must only be able to:

- access conversations they participate in
- read messages in conversations they participate in
- send messages where permitted
- modify/delete only messages they own, if modification is supported

Prevent:

- unauthorized conversation access
- cross-user message access
- IDOR vulnerabilities
- impersonation
- arbitrary sender IDs supplied by clients

The sender identity must come from the authenticated PMP identity, never from an untrusted request body.

D. MODERATION

Design a foundation for moderation without prematurely implementing the entire Stage 9 operations system.

Consider support for:

- reporting a message
- reporting a conversation
- soft moderation states
- message removal/hiding
- blocked users
- basic content moderation hooks

Keep moderation logic extensible for the future Operations stage.

Do not build a full admin dashboard yet.

Do not invent hardcoded admin credentials.

E. REALTIME TRANSPORT

The realtime architecture must remain portable.

Choose the most appropriate approach for the current architecture.

Possible approaches may include:

- Server-Sent Events
- WebSockets
- a transport abstraction that allows future replacement

The implementation must not depend on Replit infrastructure.

The architecture should allow the frontend to receive new-message events without requiring the database to be replaced.

If the existing architecture already contains a realtime abstraction, extend it rather than replacing it.

If the existing system uses an abstraction/stub, implement the real transport behind the abstraction.

Keep the transport layer separate from domain logic.
