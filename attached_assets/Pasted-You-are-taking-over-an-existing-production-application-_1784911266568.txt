CRITICAL PRODUCTION BUSINESS RULE TO VERIFY
==================================================

Verify the complete subscription-to-messaging flow.

The intended business rule is:

A user must have an active subscription before they can initiate or send messages.

Recipients should still be able to receive messages without personally being required to subscribe, depending on the current business logic.

You must verify the actual implementation rather than assuming it.

Trace the complete flow:

1. User signs up.
2. User logs in.
3. User selects a subscription plan.
4. Payment is initiated.
5. Paystack processes the payment.
6. Payment verification occurs.
7. Subscription status is written to the database.
8. Entitlements are calculated.
9. User attempts to create a conversation.
10. User attempts to send a message.
11. Backend checks subscription entitlement.
12. Message is accepted or rejected.

Confirm exactly:

- What database table stores subscriptions?
- What statuses are considered active?
- What statuses are considered inactive?
- Does a successful Paystack payment reliably create/update the subscription?
- Is payment verification server-side?
- Is the frontend allowed to influence subscription status?
- Is the backend the final authority?
- Does POST /v1/messaging/conversations require an active subscription?
- Does POST /v1/messaging/conversations/:id/messages require an active subscription?
- Can an unsubscribed user bypass the restriction by calling the API directly?
- What exact HTTP status and error code are returned?
- Does the frontend show an understandable error?
- Does the user retain their typed message if sending fails?
- Can a subscribed user successfully message another user?
- Can the recipient receive the message without a subscription?
- Are there any frontend/backend response-shape mismatches?
- Are there any Clerk authentication timing issues?
- Are there any SSR requests occurring before authentication is ready?

Do not modify anything while verifying this.
