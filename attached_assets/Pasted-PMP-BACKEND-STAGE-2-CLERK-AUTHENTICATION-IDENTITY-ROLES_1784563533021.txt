==================================================
CORE AUTHENTICATION DECISION
==================================================

Use CLERK as the sole authentication and identity provider.

Do NOT implement a parallel custom authentication system.

Do NOT create:
- Custom password hashing
- Custom password storage
- Custom login credential verification
- Custom password reset logic
- Custom refresh-token authentication system
- A second independent JWT authentication system
- A second independent session authentication system

Clerk owns:

- User registration
- User login
- Password management
- Password recovery
- Email verification
- Session lifecycle
- Authentication identity
- Clerk-issued authentication tokens

The PMP backend must verify authenticated Clerk requests.

The backend must then use the Clerk user ID to resolve the corresponding PMP application user.

The architecture must be:

Clerk
  ↓
Authenticated identity
  ↓
Clerk user ID
  ↓
PMP users table
  ↓
PMP roles
  ↓
PMP permissions
  ↓
Backend authorization

==================================================
CLERK INTEGRATION
==================================================

Integrate Clerk using a portable backend-compatible approach.

The backend must not depend on Replit.

The Clerk integration must be encapsulated behind a clean authentication abstraction.

For example, the rest of the application should be able to depend on an abstraction such as:

authenticateRequest()
or an equivalent clean interface.

The rest of the application should not be tightly coupled to Clerk-specific implementation details.

The backend must:

1. Extract the incoming Clerk authentication token/session information.
2. Verify the request using Clerk's official server-side verification mechanism.
3. Resolve the authenticated Clerk user ID.
4. Load the corresponding PMP user from PostgreSQL.
5. Attach an authenticated request context.
6. Continue to PMP role and permission authorization.

Unauthenticated requests must receive a consistent 401 response.

Authenticated users without sufficient permissions must receive a consistent 403 response.

Do not trust role information supplied by the frontend.

Do not trust arbitrary user IDs supplied by the client.

The authenticated Clerk identity must be the source of identity.

PMP PostgreSQL must be the source of application authorization.

==================================================
USER PROVISIONING
==================================================

Design a safe first-login or account-provisioning flow.

When a user authenticates with Clerk:

1. Verify the Clerk identity.
2. Search for the Clerk user ID in PMP PostgreSQL.
3. If the user exists:
   - Load the PMP user.
   - Load roles.
   - Load permissions.
4. If the user does not exist:
   - Create the appropriate PMP user record only through a controlled flow.
   - Do not allow a client to arbitrarily assign privileged roles.

The public registration flow must support the intended account types:

EMPLOYER
PROVIDER

Provider users should eventually be able to identify whether they are:

ARTISAN
or
PROFESSIONAL

Do not prematurely build the entire profile system in this stage unless required for identity provisioning.

The goal of this stage is identity and authorization foundation.
